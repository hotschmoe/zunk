# Zunk v0.8.0 -- GPU descriptor type-safety migration

**Audience**: Teak maintainers.
**Date**: 2026-04-19.
**Zunk version**: 0.8.0 (breaking; supersedes v0.7.0).
**Companion (archived)**: the zunk-handoff and zunk-roadmap docs from
`from_teak_team/` are moved to `.archive/` now that every item they
tracked is shipped. See "Prior state" below.

## TL;DR

v0.8.0 changes three GPU descriptor types from `u32`-with-`// enum-name`
comments to typed enum fields. ABI is unchanged, but call sites that
used the `init(...)` constructors must migrate to struct literals (or to
`fromSlice` for one case). Two-digit line change on Teak's side.

Tag to pin once this is merged: **v0.8.0**.

## Why this changed

While reviewing v0.7.0 we noticed `SamplerDescriptor.init` silently
aliased `address_w` to the caller's `u` parameter (no `w` argument
existed). The type system didn't catch it because every enum field was
stored as `u32` with the enum name relegated to a marker comment:

```zig
mag_filter: u32 = 0, // FilterMode
address_u: u32 = 0,  // AddressMode
```

That pattern also required matching `@intFromEnum` calls in every `init`
body, which is exactly where the aliasing bug lived:

```zig
.address_u = @intFromEnum(u),
.address_v = @intFromEnum(v),
.address_w = @intFromEnum(u), // <-- silently re-uses `u`
```

Rather than add a `w` parameter and leave the `u32 + // comment`
convention in place (one-site fix, pattern intact), we migrated the
three pure-enum descriptors to typed fields with enum defaults. The
`// FilterMode` comments are gone because the type says it; the
`@intFromEnum` calls are gone because callers pass enum values directly;
and the `init` factories are gone for the two descriptors where a
struct literal reads cleaner.

`BindGroupLayoutEntry` and `BindGroupEntry` were intentionally left
alone: their `type_variant` / `entry_type` fields are genuine union
discriminants (their meaning shifts based on a sibling field), so
`initBuffer` / `initTexture` / `initSampler` factories remain the right
API there.

## API delta

### SamplerDescriptor

Before (v0.7.0):
```zig
const sampler = gpu.createSampler(gpu.SamplerDescriptor.init(
    .linear,        // mag
    .linear,        // min
    .clamp_to_edge, // address_u (also silently copied to address_w)
    .clamp_to_edge, // address_v
));
```

After (v0.8.0):
```zig
const sampler = gpu.createSampler(.{
    .mag_filter = .linear,
    .min_filter = .linear,
});
```

Notes:
- Struct literal is inferred from `createSampler`'s parameter type.
- `address_u` / `address_v` / `address_w` default to `.clamp_to_edge` --
  override any of the three individually, including `address_w` (which
  was unreachable in v0.7.0).
- `mag_filter` / `min_filter` default to `.nearest`.
- `.init` is deleted. Struct-literal is the only constructor.

### VertexAttribute

Before:
```zig
gpu.VertexAttribute.init(0, .float32x2, 0)
```

After:
```zig
.{ .shader_location = 0, .format = .float32x2, .offset = 0 }
```

Field order inside the struct is `format, offset, shader_location,
_padding` (ABI-locked; unchanged). Struct-literal form lets you use
named fields so call-site order is yours.

### VertexBufferLayout

Before:
```zig
gpu.VertexBufferLayout.init(@sizeOf(Vertex), .vertex, &attrs)
```

After:
```zig
gpu.VertexBufferLayout.fromSlice(@sizeOf(Vertex), .vertex, &attrs)
```

Renamed `init` -> `fromSlice` because it really does translate a Zig
slice into `(attributes_ptr, attributes_len)` -- not trivial sugar, so
kept as a helper. Internally it now stores `step_mode` as a
`VertexStepMode` value directly (no `@intFromEnum` on the caller's
side).

If you prefer the struct-literal form:
```zig
gpu.VertexBufferLayout{
    .array_stride = @sizeOf(Vertex),
    .step_mode = .vertex,
    .attributes_ptr = @intFromPtr(attrs.ptr),
    .attributes_len = @intCast(attrs.len),
}
```

`fromSlice` is strictly ergonomic -- use whichever reads better.

## ABI guarantees (nothing changed)

- `SamplerDescriptor`: 24 bytes, same field order.
- `VertexAttribute`: 16 bytes, same field order.
- `VertexBufferLayout`: 16 bytes, same field order.
- All `enum(u32)` declarations have identical layout and ordinal values
  to their v0.7.0 `u32`-with-comment counterparts.
- Struct-size tests in `src/web/gpu.zig` still guard the JS DataView
  reader contract.

Concretely: if you kept sending the old bit patterns to
`zunk_gpu_create_sampler` / `create_render_pipeline`, nothing would
break -- the wasm import side still receives the same 24 / 16 bytes.
The migration is purely at the Zig construction layer.

## Migration checklist for Teak

Search-and-replace scope in Teak:

1. `SamplerDescriptor.init(` -- replace with struct literal at
   `createSampler` call sites. `address_w` is now reachable; set it
   explicitly if you need it, otherwise defaults to `.clamp_to_edge`.
2. `VertexAttribute.init(loc, fmt, off)` -- replace with
   `.{ .shader_location = loc, .format = fmt, .offset = off }` (or
   positional struct literal if you prefer -- field order is
   `format, offset, shader_location, _padding`).
3. `VertexBufferLayout.init(` -- rename to `.fromSlice(` (same args).

Then bump the pinned tag in `build.zig.zon` to `v0.8.0`.

Five scoped greps cover it. Zunk's own examples took three edits total
(vertex-triangle, texture-demo, text-demo).
